BANNIN — HOW TO USE IT
=======================

BANNIN is a tool that scans your code for security problems (bugs,
leaked passwords, vulnerable dependencies, misconfigured infra) and
shows you what it found — either as a report file, or in a web
dashboard.

FASTEST WAY TO TRY IT (2 commands)
------------------------------------
    ./scripts/install-tools.sh     (installs the scanner tools)
    ./scripts/start-web.sh         (builds + starts everything, opens the dashboard)

That's it for a local look. For real CI/CD or GUI setup, read on.

There are two ways to use it. Pick one, or use both.


1) CI/CD — runs automatically on every build, fails the build if bad
   -----------------------------------------------------------------

   Use this when you want every pull request / build to be checked
   automatically, and to BLOCK the build if something serious is found.

   Steps:

   a) Add a config file called bannin.yaml to your repo (copy
      configs/bannin.example.yaml and edit the "target" if needed).

   b) In your CI pipeline, install the scanner tools once per job:

          ./scripts/install-tools.sh --ci --skip-zap

      (--ci = no interactive prompts; --skip-zap because ZAP scans a
      running app, not source code — only add it to a job that has one
      deployed. Drop --skip-zap if you do have a test deployment.)

   c) Add a step that runs:

          bannin scan --config bannin.yaml

      That's it. BANNIN will:
        - scan your code
        - print a summary in the CI logs
        - save a report (report.html and report.json) as build output
        - FAIL the build (non-zero exit code) if it finds anything at
          or above the severity you set in "fail_on_severity"

   d) Download/open report.html from your CI's build artifacts to see
      full details of everything it found, grouped by category.

   Nothing else needed. No dashboard, no server — just the scan step.


2) STANDALONE / GUI — a dashboard you and your team can browse
   -------------------------------------------------------------

   Use this when you want a live web page showing scan results,
   history over time, and details you can click into — instead of
   reading a report file.

   Shortcut: run ./scripts/start-web.sh and skip straight to step (e) —
   it does (a)-(d) for you and opens your browser. Manual steps, if you
   want to see each part:

   a) Same as above: have a bannin.yaml with:
          storage:
            driver: "sqlite"
            dsn: "./bannin.db"
      (this makes BANNIN remember every past scan, not just the last one)

   b) Run a scan at least once:
          bannin scan --config bannin.yaml

   c) Build the dashboard once (produces web/dist):
          cd web
          npm install
          npm run build
          cd ..

   d) Start BANNIN — it serves the dashboard AND the data on one port:
          bannin serve --config bannin.yaml

   e) Open the address it prints (usually http://127.0.0.1:8080) in
      your browser. You'll see:
        - a security score for the latest scan
        - findings broken down by severity and category
        - a searchable table of every finding, click one for full detail
        - scan history over time
        - whether each scanner (Semgrep, Trivy, etc.) ran OK

   f) To get new data in the dashboard, just run "bannin scan" again —
      the dashboard picks up the new scan automatically (refresh the
      page).

   g) Or scan something new WITHOUT touching a terminal at all: click
      "New Scan" in the sidebar, paste a target (a URL like
      https://example.com for a live app, or a local folder path),
      pick scanners if you want (leave blank to auto-pick), click
      "Run scan". It runs in the background and the page shows the
      scanner's live activity (spider/active-scan phases) and elapsed
      time while it works, then a "View results" button when it's done.

   ZAP scan depth: for URL targets, set zap.mode in bannin.yaml to
   "quick" (default — fast, mostly finds header/config issues) or
   "full" (runs ZAP's full active-scan policy to probe for injection
   vulnerabilities like SQLi/XSS — much deeper, much slower, and far
   more aggressive, so only point it at systems you're allowed to
   actively attack).


SCANNING A REAL WEB APP (JS + LOGIN)
-------------------------------------

   Most real apps are a JavaScript front end with everything behind a
   login. Two settings handle that (both full mode only):

   1) JavaScript apps — turn on the AJAX spider:

          zap:
            mode: "full"
            ajax: true

      The normal spider only reads links out of static HTML, so a
      single-page app looks nearly empty to it. The AJAX spider drives
      a real headless browser instead. (Needs Chrome/Chromium/Firefox —
      install-tools.sh sets one up.)

   2) Pages behind a login — configure auth:

          zap:
            mode: "full"
            auth:
              method: "form"     # form | json | header | browser
              login_url: "https://your-app/login"
              username: "tester"
              username_field: "username"   # your form's field names
              password_field: "password"
              logged_in_regex: "Logout"    # how ZAP knows it's logged in

      Which method:
        form    - normal login form (POST username + password)
        json    - login endpoint taking a JSON body (SPA/API backends)
        header  - static token header, e.g. Authorization: Bearer xxx
        browser - headless browser fills the login form (JS-heavy apps)

      PASSWORDS/TOKENS ARE SECRETS. Do not put them in bannin.yaml.
      Pass them as environment variables:

          export BANNIN_ZAP_AUTH_USERNAME=tester
          export BANNIN_ZAP_AUTH_PASSWORD='your-password'
          export BANNIN_ZAP_AUTH_TOKEN='Bearer eyJ...'   # method: header

      Set logged_in_regex (some text only logged-in pages show, like
      "Logout") — that's how ZAP notices the session expired and logs
      back in. Without it, a scan can silently end up testing only
      logged-out pages.


THE SHORT VERSION
------------------
- CI/CD  = "bannin scan" in your pipeline, fails the build on bad findings.
- GUI    = "bannin scan" to gather data + "bannin serve" (which serves
           the dashboard and the data together on one port) to browse it.
- You can do both at once: same scans feed both the CI gate and the
  dashboard, as long as storage.driver is set to sqlite.
